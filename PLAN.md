# Twitch Media Widget — план имплементации

Аналог MemAlerts: зритель отправляет произвольный медиафайл (картинка, гифка, видео, аудио)
на стрим, файл проигрывается в OBS-оверлее с ограниченной длительностью.

## Зафиксированные решения

- **Стек:** TypeScript везде (Node.js бэкенд + React фронт).
- **Монетизация:** MVP бесплатный (лимиты + кулдауны), донаты — отдельной фазой позже.
- **Модерация:** гибрид — зрители из белого списка идут в автопоказ, остальные в очередь модерации.
- **Хостинг:** пока локально; архитектура не должна мешать переезду в облако.

## Архитектура

```
┌─────────────┐  upload   ┌──────────────────────┐  WS push   ┌────────────┐
│ Страница    │──────────▶│ Бэкенд (Fastify)     │───────────▶│ Оверлей    │
│ зрителя     │           │  - Twitch OAuth      │            │ (OBS       │
│ /c/<канал>  │           │  - ffprobe/ffmpeg    │◀───────────│  Browser   │
└─────────────┘           │  - SQLite + Drizzle  │  "done"    │  Source)   │
                          │  - очередь показа    │            └────────────┘
┌─────────────┐           │  - файлы на диске    │
│ Дашборд     │◀─────────▶│                      │
│ стримера    │  REST+WS  └──────────────────────┘
└─────────────┘
```

- **Оверлей** аутентифицируется секретным токеном в URL (`/overlay?token=...`),
  как у StreamElements — OAuth в Browser Source невозможен.
- **Двойное ограничение длительности:** сервер обрезает/отклоняет файл по ffprobe,
  оверлей дополнительно снимает медиа с экрана по таймеру.
- **Очередь показа** per-channel: оверлей проигрывает строго по одному,
  по событию `playback:done` сервер выдаёт следующий элемент.

## Структура монорепо (pnpm workspaces)

```
apps/
  server/    — Fastify + Socket.IO + Drizzle + ffmpeg
  web/       — React: страница зрителя + дашборд стримера
  overlay/   — лёгкая страница для OBS Browser Source
packages/
  shared/    — общие типы: события WS, DTO, zod-схемы
```

## Модель данных (первый набросок)

- `users` — twitch_id, login, display_name, avatar; роль вычисляется по контексту.
- `channels` — владелец (user), overlay_token, настройки (макс. длительность,
  макс. размер, громкость, кулдаун, включён ли приём).
- `whitelist` — (channel_id, user_id), кто в автопоказе.
- `bans` — (channel_id, user_id).
- `submissions` — channel_id, sender_id, путь к файлу, mime, длительность,
  статус: `pending | approved | rejected | played | expired`.

### Хранение файлов: эфемерное

Файл живёт только от загрузки до показа. В терминальных статусах
(`played | rejected | expired`) файл удаляется, запись в БД остаётся как история
(метаданные: кто, когда, mime, длительность — без самого медиа).

- Удаление не мгновенное: фоновая зачистка раз в несколько минут удаляет файлы,
  пробывшие в терминальном статусе дольше ~10 минут — чтобы сервер мог переиграть
  текущий элемент, если оверлей переподключился посреди показа.
- `pending` дольше N часов → переводится в `expired` той же зачисткой.
- Следствия: история показов — без превью; «библиотека одобренных мемов» канала
  несовместима с эфемерным хранением (отложено до пост-MVP).

## Фазы

### Фаза 0 — Скелет проекта ✅ (готово)
- pnpm workspaces, TypeScript-конфиги, ESLint + Prettier.
- `apps/server`: Fastify «hello world», подключение SQLite через Drizzle, миграции.
- `apps/web` и `apps/overlay`: пустые Vite-приложения.
- `packages/shared`: общий пакет типов, импортируется и сервером, и фронтами.
- Скрипт `dev`, поднимающий всё разом (turbo или concurrently).

**Готово, когда:** `pnpm dev` поднимает сервер и оба фронта, сервер пишет/читает SQLite.

### Фаза 1 — Сквозной поток без авторизации (tracer bullet) ✅ (готово)
Один захардкоженный канал, никакого логина — цель прогнать файл от загрузки до экрана.
- Эндпоинт загрузки (multipart): валидация mime (whitelist: jpg/png/gif/webp/mp4/webm/mp3/ogg)
  и размера, magic-bytes-проверка содержимого, не доверять расширению.
- ffprobe: измерить длительность; видео/аудио длиннее лимита — обрезать ffmpeg'ом
  (или отклонять — настройка).
- Сохранение на диск через интерфейс `Storage` (чтобы потом подменить на S3).
- Socket.IO: комната канала, событие `media:play` с URL и длительностью.
- Оверлей: прозрачный фон, проигрывание image/video/audio, жёсткий таймер снятия,
  очередь — по одному, `playback:done` серверу.
- Жизненный цикл файла: после `playback:done` сабмишен помечается `played`,
  фоновая зачистка удаляет файл (см. «Хранение файлов: эфемерное»).
- Простейшая форма загрузки на странице зрителя.

**Готово, когда:** файл, загруженный через форму, проигрывается в OBS и исчезает по таймеру.

### Фаза 2 — Twitch OAuth и мультиканальность ✅ (готово; реальный OAuth ждёт ключей в apps/server/.env, для локалки есть fake-режим)
- OAuth-флоу через Twurple: логин стримера и зрителя (одна и та же учётка Twitch,
  разница только в контексте).
- Сессии (cookie + signed session или JWT).
- Регистрация канала: стример логинится → создаётся `channel` + overlay_token.
- Публичная страница зрителя `/c/<login>`: логин обязателен для отправки.
- Оверлей привязан к каналу по токену, кнопка «перевыпустить токен» у стримера.

**Готово, когда:** два разных стримера могут завести каналы, и медиа попадает только
в нужный оверлей.

### Фаза 3 — Модерация и белый список ✅ (готово)
- Отправка от зрителя из белого списка → сразу `approved` → в очередь показа.
- Остальные → `pending` → дашборд: превью медиа, кнопки «одобрить / одобрить и в белый
  список / отклонить / забанить».
- Live-обновление очереди модерации по WebSocket.
- Кулдаун на зрителя и на канал, лимит отправок в час.
- Баны: молчаливое отклонение.

**Готово, когда:** новичок ждёт одобрения, одобренный зритель попадает на экран сразу,
забаненный — никогда.

### Фаза 4 — Настройки и удобство стримера ✅ (готово; позиция/размер оверлея на экране — отложено)
- Настройки канала в дашборде: макс. длительность, макс. размер файла, громкость,
  позиция/размер на экране, пауза приёма («стоп-кран»).
- Панель «сейчас играет» + кнопка скипа (мгновенно гасит оверлей).
- Тестовая отправка из дашборда.
- История показов (метаданные без превью — файлы к этому моменту удалены).
- Имя отправителя поверх медиа в оверлее (опционально, настройкой).

### Фаза 5 — Надёжность и безопасность ✅ (готово)
- ✅ Перекодирование: видео → mp4 (h264+aac, ≤720p/30fps), аудио → mp3,
  картинки и гифки → webp (sharp). Обрезка, чистка метаданных и защита
  от экзотических кодеков одним проходом.
- ✅ Rate limiting: глобальный потолок на IP + отдельный жёсткий на аплоад
  (RATE_LIMIT_GLOBAL / RATE_LIMIT_UPLOAD).
- ✅ Восстановление оверлея после разрыва WS (replay текущего элемента) — с фазы 1.
- ✅ Логирование pino — с фазы 0. Метрики — отложены (для пет-проекта хватает логов).

### Фаза 6 — Деплой и монетизация (после MVP)

Хостинг выбран: **Render free (Docker) + Turso (БД) + UptimeRobot (автопинг /api/ping)**.

Подготовка кода готова ✅:
- БД переведена на @libsql/client: локально file:SQLite, в проде Turso (env
  TURSO_DATABASE_URL/TURSO_AUTH_TOKEN). Эфемерный диск Render не страшен:
  медиа эфемерное по дизайну, БД снаружи.
- Прод-режим (NODE_ENV=production или SERVE_STATIC=1): сервер сам раздаёт
  собранные web (/, SPA-fallback) и overlay (/overlay/) — один домен, без CORS.
- Dockerfile (node:24-slim + ffmpeg), render.yaml (Blueprint), /api/ping для пинга.
- trustProxy + secure-cookies в проде.

Осталось (руками, аккаунты): GitHub-репозиторий, Turso DB, Render Blueprint,
UptimeRobot-монитор, второй redirect URL в Twitch-консоли.

Монетизация (потом):
- Интеграция донатов: DonationAlerts API или Stripe — донат как условие показа
  либо приоритет в очереди.
- Идея: триггер через Twitch Channel Points (EventSub) как «бесплатная валюта».

## Отложено (пост-MVP)

- «Библиотека» одобренных мемов канала для повторной отправки — требует постоянного
  хранения файлов, несовместима с эфемерной моделью MVP.
